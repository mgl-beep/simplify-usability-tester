<project_specification>
  <project_name>SIMPLIFY — Canvas LMS Accessibility &amp; Quality Scanner</project_name>

  <overview>
    SIMPLIFY is a web application for California Community College faculty and instructional designers
    that scans Canvas LMS courses for accessibility violations and quality rubric compliance issues,
    suggests AI-powered fixes, and publishes approved corrections back to Canvas.

    The app targets three standards frameworks simultaneously:
    - CVC-OEI Course Design Rubric (Sections A–D) — California Virtual Campus accessibility standard
    - Quality Matters 7th Edition (Standards 1–8) — national online course quality benchmark
    - Peralta Online Equity Rubric v3.0 (Standards E1–E8) — equity-focused design standard

    Core workflow: Connect Canvas → Select Course → Scan → Review Issues → Stage Fix → Publish to Canvas

    The application runs entirely in the browser (React SPA). All Canvas API calls are proxied through
    a Supabase Edge Function to bypass CORS. Authentication is token-based (Canvas Personal Access Token).
    There is no native Canvas LTI integration yet.

    Live URL: https://simplify-lti.vercel.app
  </overview>

  <architecture>
    <state_management>
      All global state lives in App.tsx (~1,800 lines). Key state variables:
      - selectedCourse: {courseId, courseName, isImported} — the active course
      - scanResults: ScanIssue[] — current course's detected issues
      - allCourseScanResults: Record&lt;number, ScanIssue[]&gt; — per-course cached results
      - isScanning: boolean — scan in progress flag
      - enabledStandards: string[] — active rubric filters (cvc-oei, qm, peralta)
      - aiSuggestionsCache: Record&lt;string, string&gt; — cached AI suggestions by element hash
      - currentView: string — active page ("simplify" | "courses" | "analytics" | etc.)

      Props drill from App.tsx → SimplifyDashboard → ScanPanel, Analytics, IssueDetailModal, etc.
      No external state management library is used.
    </state_management>

    <data_flow>
      1. Browser (React SPA) sends Canvas API requests to Supabase Edge Function
      2. Edge Function forwards with Authorization: Bearer canvas_token to Canvas LMS API
      3. Canvas returns HTML content
      4. Edge Function relays response to browser
      5. Browser parses HTML with DOMParser, runs scanners locally
      6. ScanIssue[] stored in React state + localStorage cache
      7. Fix staged locally (fixStaging.ts stores original + fixed HTML)
      8. User reviews Before/After preview in IssueDetailModal
      9. User confirms → canvasFixer.ts → Supabase Edge Function → Canvas API (PUT/POST)
    </data_flow>

    <scan_engine>
      courseScanner.ts orchestrates five scanners sequentially:
      1. accessibilityScanner.ts — WCAG 2.0 AA: alt text, headings, tables, contrast, lists
      2. usabilityScanner.ts — CVC-OEI usability: captions, chunking, nav depth, content length
      3. cvcOeiRubricScanner.ts — full CVC-OEI Sections A–D rubric (~80KB, largest scanner)
      4. designScanner.ts — visual consistency: font families, styling patterns
      5. linkDetector.ts — broken links, long URLs, unclear link text ("click here")

      Each scanner is a pure function: takes HTML string → returns ScanIssue[].
      standardsMapping.ts tags each issue with rubric codes after scanning.
    </scan_engine>

    <fix_engine>
      Two-phase workflow enforced by fixStaging.ts:
      Phase 1 — Stage: Generate fixed HTML locally, store original for undo, set status='staged'
      Phase 2 — Publish: User confirms → canvasFixer.ts sends fixed HTML through Supabase proxy to Canvas API

      canvasFixer.ts (~63KB) handles every fix type with dedicated cases per issue category.
      Each issue category (alt-text, contrast, heading, table, link-text, objectives) has its own
      rendering branch in IssueDetailModal.tsx (~186KB).
    </fix_engine>

    <ai_integration>
      All AI calls route through aiFixerService.ts → Supabase Edge Function → OpenAI GPT-4o.
      Supported AI features:
      - Alt text generation (GPT-4o Vision — analyzes actual images)
      - Table caption generation
      - Link text suggestions (descriptive replacements for "click here")
      - Learning objectives generation (Bloom's taxonomy aligned)
      - Content rewriting (for dense paragraphs)
      - Content template generation (for course builders)

      AI suggestions are cached by element hash to avoid duplicate API calls.
      AI NEVER publishes without explicit user confirmation.
    </ai_integration>

    <persistence>
      All scan data persists to localStorage — no cloud sync for scan data:
      - simplify_all_scan_results — ScanIssue[] per course, keyed by courseId
      - simplify_all_scan_times — last scan timestamps per course
      - simplify_enabled_standards — active rubric filter state
      - canvasToken / canvasDomain — Canvas connection credentials

      Guard: courseId=0 ("All Courses" view) is explicitly excluded from localStorage writes.
    </persistence>
  </architecture>

  <scan_issue_schema>
    Every detected issue conforms to this TypeScript interface:

    interface ScanIssue {
      id: string;                      // Unique identifier
      type: string;                    // e.g., 'missing-alt-text', 'contrast', 'broken-link'
      category: string;                // 'accessibility' | 'usability' | 'design' | 'content'
      severity: 'high' | 'medium' | 'low';
      title: string;                   // Human-readable issue title
      description: string;             // Explanation of the problem
      location: string;                // Human-readable location (e.g., "Week 1 Overview")
      contentType: string;             // 'page' | 'assignment' | 'announcement' | 'discussion'
      contentId: string | number;      // Canvas content ID for API updates
      courseId?: number;                // Which course this belongs to
      autoFixAvailable: boolean;       // Whether an automated fix can be generated
      standardsTags: string[];         // e.g., ['cvc-oei:D7', 'qm:8.1', 'peralta:E3']
      status: 'pending' | 'staged' | 'published' | 'ignored' | 'resolved';
      elementHtml?: string;            // Raw HTML snippet that triggered the issue
      textColor?: string;              // Detected text color (hex) for contrast issues
      backgroundColor?: string;        // Effective background color (hex) for contrast issues
      suggestedFix?: string;           // AI-generated or auto-generated fix suggestion
      fixSteps?: string[];             // Step-by-step remediation instructions
      originalContent?: string;        // Stored on staging for undo capability
      whereToAddPageUrl?: string;      // For objectives issues: specific Canvas page URL
      stagedFix?: object;              // Populated when fix is staged but not published
    }

    Standards tags format: always lowercase prefix, colon separator, uppercase section ID.
    Examples: "cvc-oei:D5", "qm:8.3", "peralta:E4"

    Severity scoring: high = -10 pts, medium = -5 pts, low = -2 pts from a base of 100. Floor at 0.

    Status state machine: pending → staged → published | ignored | resolved
  </scan_issue_schema>

    <accessibility_scanning>
      CVC-OEI Section D / WCAG 2.0 AA checks:

      - D7 / WCAG 1.1.1: Missing alt text on images (detects missing or empty alt on non-decorative images)
      - D1: Heading hierarchy violations (skipped levels, missing H1, bold-as-heading patterns)
      - D4: Tables without th headers (checks first row for th presence)
      - D4: Tables without caption element (checks for caption on data tables)
      - D5 / WCAG 1.4.3: Color contrast below 4.5:1 ratio (inline style + data-mce-style detection)
      - D5: Background color detection (walks ancestor chain: style, data-mce-style, bgcolor attributes)
      - D3: Unclear link text ("click here", "read more", generic phrases)
      - D3: Raw URL as link text (URLs over 50 characters used as display text)
      - D12: Video without captions (YouTube cc_load_policy check — partial coverage)

      Not yet implemented:
      - D2: Lists created with manual dashes/asterisks instead of ul/ol markup
      - D6: Color as sole information conveyor
      - D13: Audio without transcript
      - D16: Auto-playing audio/video (autoplay attribute check)
      - D14: Flashing content (out of scope v1)
      - D8: PDF reading order (out of scope v1)
      - D9: Slide accessibility (out of scope v1)
      - D10: Spreadsheet accessibility (out of scope v1)
    </accessibility_scanning>

    <rubric_scanning>
      CVC-OEI Sections A–C checks:

      - A1/A2: Learning objectives present in modules (detection exists; AI generation works)

    <fix_workflow>
      Two-phase stage → review → publish workflow:

      - Stage fix: Generate fixed HTML locally, preview without touching Canvas
      - Before/After HTML preview: Side-by-side comparison in IssueDetailModal
      - Publish fix: Confirmed fix sent to Canvas via Supabase proxy
      - Undo published fix: Original content stored for revert
      - Batch stage: Stage all auto-fixable issues at once
      - Batch publish: Publish all staged fixes (partial — lacks per-item progress)

      Per-issue-type fixes:
      - Alt text: Manual entry or AI-generated (GPT-4o Vision)
      - Heading hierarchy: Level reassignment or conversion to paragraph
      - Table headers: td → th conversion in first row
      - Table caption: Manual entry or AI-generated
      - Color contrast: Inline color style override with WCAG-passing color
      - Link text: Manual entry or AI-generated descriptive replacement
      - Learning objectives: AI generation (Bloom's taxonomy) injected into module page

      Not yet implemented:
      - Dense paragraph AI rewrite (flagged but no AI suggestion in fix flow)
      - Layout table semantic conversion (flagged but no purpose-selection step)
      - AI broken link replacement (flagged but no replacement URL suggestion)
    </fix_workflow>

  <invariants>
    These rules MUST NOT be violated. Any feature implementation must preserve these:

    1. STAGE → REVIEW → PUBLISH WORKFLOW: Fixes must never be written to Canvas without explicit
       instructor confirmation. Never add auto-publish shortcuts. Every fix goes through fixStaging.ts
       before the user explicitly publishes.

    2. AI NEVER PUBLISHES WITHOUT CONFIRMATION: All AI-generated content (alt text, captions,
       objectives, link text, rewrites) must go through edit-and-confirm in IssueDetailModal.

    3. SUPABASE PROXY FOR ALL CANVAS API CALLS: Canvas does not allow direct browser requests (CORS).
       Never call Canvas API directly from frontend code.

    4. STANDARDS TAGS ON EVERY SCAN ISSUE: Every issue from any scanner must include
       standardsTags: string[]. Format: 'cvc-oei:D7', 'qm:8.3', 'peralta:E3' (lowercase prefix,
       colon, uppercase section ID).

    5. COLOR VALUES ON CONTRAST ISSUES: textColor and backgroundColor (hex) must be present on
       contrast issues. IssueDetailModal reads these directly.

    6. COURSEID GUARD FOR LOCALSTORAGE: allCourseScanResults cache must never write with courseId=0.
       The guard in App.tsx must be preserved.

    7. SCANNERS ARE PURE FUNCTIONS: Each scanner takes HTML string, returns ScanIssue[].
       No side effects, no API calls inside scanners.

    8. AI CALLS THROUGH SERVICE LAYER: All AI requests go through aiFixerService.ts — never add
       direct fetch calls to AI endpoints in components.

    9. STATUS STATE MACHINE: ScanIssue.status follows: pending → staged → published | ignored | resolved.
       Do not introduce additional status values.

    10. SKIPFETCH GUARD IN ISSUEDETAILMODAL: Issues with contentType 'module' or 'course',
        contentId 'syllabus', or categories 'policies'/'objectives' must set skipFetch=true
        before calling loadContent().
  </invariants>
